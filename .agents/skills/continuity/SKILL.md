# Continuity — перевірка консистентності

## Контекст

Цей скіл аналізує рукопис на наявність помилок, протиріч і нестиковок. **Не редагує файли** — лише звітує про знайдені проблеми з точними вказівками на місце і пропозиціями виправлення.

### Релевантні директорії

Усі директорії проєкту: `scenes/`, `chapters/`, `characters/`, `world/`, `events/`, `journal/`, `inbox/`, `notes/`

### Ключові команди

```bash
zk orphans               # нотатки без вхідних посилань
zk backlinks <slug>      # зворотні посилання на нотатку
just search "запит"      # пошук по всіх md-файлах
just stats               # загальна статистика
just list-drafts         # сцени/глави з status: draft
just wordcount           # оновити word-count у сценах
```

## Процес

### Крок 1: Визначити обсяг перевірки

Запитай або визнач з контексту, що саме перевіряти:

- **Конкретний персонаж:** `just search "ім'я"` + аналіз усіх згадок
- **Конкретна глава:** прочитати главу + всі сцени в ній
- **Увесь рукопис:** повна перевірка (довше, але вичерпно)
- **Тільки wikilinks:** перевірка мертвих і пропущених посилань

### Крок 2: Перевірка wikilinks

1. **Мертві wikilinks** — знайди всі `[[...]]` і перевір, чи існує файл:

```bash
for f in */*.md; do
  grep -oP '!?\[\[\K[^\]]+' "$f" | while read slug; do
    # перевірити чи існує файл з цим slug у будь-якій директорії
  done
done
```

2. **Пропущені wikilinks** — знайди згадки персонажів (з `characters/`) і локацій (з `world/`) у тексті сцен, які не обгорнуті в `[[...]]`.

### Крок 3: Перевірка персонажів

Для кожного персонажа:

1. Прочитай `characters/<slug>.md` — зафіксуй ключові атрибути:
   - **Ім'я** — чи всюди однаково?
   - **Зовнішність** — очі, волосся, зріст, особливі прикмети
   - **Роль** — чи відповідає поведінка в сценах заявленій ролі?

2. Знайди всі згадки в сценах через `zk backlinks <slug>`

3. Для кожної сцени порівняй опис персонажа з карткою:
   - Чи не змінився колір очей між сценами?
   - Чи не з'явилися суперечливі факти біографії?

### Крок 4: Перевірка хронології подій

1. Знайди всі події через `zk list --path events`
2. Прочитай `date` у frontmatter кожної події
3. Відсортуй хронологічно
4. Перевір сцени, що посилаються на події — чи не порушується порядок?
   - Сцена 1: «після пожежі» → Сцена 2: «під час пожежі» → **помилка хронології**

5. Перевір `timeline` у frontmatter сцен — чи не суперечить одна одній?

### Крок 5: Перевірка локацій

1. Для кожної локації з `world/` — знайди сцени, де вона згадується
2. Порівняй опис локації в картці vs опис у сценах:
   - «Старий млин» — в картці: дерев'яний, у сцені: кам'яний → **помилка**
   - «Темний ліс» — в картці: хвойний, у сцені: дубовий гай → **помилка**

### Крок 6: Сформувати звіт

Для кожної проблеми вкажи:

```markdown
### [error] / [warn] — короткий опис

**Файл:** `scenes/ch02-sc03.md:15` (файл:рядок)
**Деталі:** що саме не так
**Контекст:** що очікувалося vs що знайдено
**Пропозиція:** як виправити
```

**Рівні проблем:**
- `[error]` — фактична помилка, яку треба виправити (dead link, протиріччя)
- `[warn]` — потенційна проблема (пропущений wikilink, неоднозначність)

## Правила

1. **Тільки звіт, без редагування.** Цей скіл не змінює файли. Якщо користувач хоче виправити проблеми — він має явно попросити або переключитися на відповідний скіл.
2. **Точні координати.** Для кожної проблеми вказуй шлях до файлу. Якщо можливо — номер рядка.
3. **Контекст для виправлення.** Не просто «знайдено помилку», а «ось що не так, ось як має бути, ось де це виправити».
4. **Групуй проблеми** за типом (wikilinks / персонажі / хронологія / локації) для зручності читання.
5. **Не панікуй через дрібниці.** `[warn]` для незначних розбіжностей, `[error]` для критичних.
6. **Перед перевіркою запусти `just wordcount`** — це оновить метадані і виявить порожні сцени.

## Приклади

### Приклад 1: Перевірка персонажа

> **Користувач:** Перевір чи все ок з [[vlad]].

**Дії агента:**
1. Читає `characters/vlad.md` → фіксує: очі сині, волосся чорне, зріст високий
2. Запускає `zk backlinks vlad` → знаходить 8 сцен
3. Читає кожну сцену, шукає описи Влада
4. Знаходить у `ch03-sc02`: «зелені очі Влада звузилися»
5. Формує звіт:
   ```markdown
   ### [error] Невідповідність зовнішності [[vlad]]

   **Файл:** `scenes/ch03-sc02.md` (рядок: очі)
   **Деталі:** У картці персонажа очі сині, у сцені — зелені.
   **Пропозиція:** Виправити на «сині» у `scenes/ch03-sc02.md` або змінити картку.
   ```

### Приклад 2: Повна перевірка wikilinks

> **Користувач:** Перевір усі wikilinks у рукописі.

**Дії агента:**
1. Збирає всі `[[...]]` з усіх md-файлів
2. Для кожного перевіряє існування файлу
3. Знаходить мертві: `[[the-old-man]]` — файл не знайдено
4. Перевіряє пропущені: імена персонажів без wikilinks у сценах
5. Формує звіт:
   ```markdown
   ## Wikilinks — перевірка

   ### [error] Мертві wikilinks (2)
   - `[[the-old-man]]` у `scenes/ch01-sc01.md` — файл не знайдено. Можливо, `[[myron]]`?
   - `[[dark-castle]]` у `scenes/ch03-sc05.md` — файл не знайдено.

   ### [warn] Пропущені wikilinks (3)
   - «Влад» у `scenes/ch02-sc03.md:5` — `[[vlad]]` існує, wikilink не додано.
   - «старий млин» у `scenes/ch01-sc02.md:12` — `[[old-mill]]` існує, wikilink не додано.
   ```

### Приклад 3: Хронологія

> **Користувач:** Перевір хронологію подій у главі 2.

**Дії агента:**
1. Читає `chapters/ch02-*.md` → знаходить список сцен через `![[...]]`
2. Читає `timeline` кожної сцени
3. Виявляє: сцена 2 — «за день до пожежі», сцена 3 — «пожежа», сцена 1 — «після пожежі» → **помилка порядку**
4. Звітує з пропозицією змінити порядок сцен у главі
